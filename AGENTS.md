## Recent Learnings

- **Rehydrated no-VAD runs can hang on a single turn with a zero-byte transcript -> When all other `turn_runs/` are complete but one `transcript.jsonl` is size 0 and still streaming in `run.log`, finalize the parent run by treating that turn as an explicit failed/empty-response row instead of dropping it or waiting indefinitely -> Preserves a 75-turn judged artifact set and makes the runtime failure visible in `failed_turns`, as seen on conversation-bench synthetic turn 67 and real-audio person1 turn 36 on March 18, 2026.**
- **Multi-call turn specs need list-shaped responses on every affected turn -> When `required_function_call` is a list, keep `function_call_response` as a same-length list and add a regression assertion over the concrete benchmark turns that use multi-call updates -> Prevents harness-level `scripted tool responses are inconsistent` failures from benchmark data shape drift, as seen in `appointment_bench` turn 21.**
- **Deep error analysis packet filenames collide across parallel benchmark workers -> When running `.agents/skills/model-run-error-analysis/scripts/deep_error_analysis.py` in parallel, copy `results/failure_review_rows.csv` and `results/failure_review_rows.jsonl` to benchmark-and-run-specific filenames immediately after each invocation -> The helper script honors a unique markdown `--output` path but still writes shared packet filenames, so concurrent analyses can overwrite one another.**
- **Rehydrated judge prompts should use golden prior history, not prior transcript outputs -> When judging `--rehydrate` runs, feed the judge a hydrated golden-history section plus current-turn actual outputs, and explicitly forbid using one target turn's actual block as state for another -> Prevents late long-range-memory turns from inheriting earlier real transcript tool failures that should be overridden by golden prior context.**
- **Prompt-visible catalog names plus hidden tool-only details is a cleaner grocery contract -> Expose a names-only product index and confusable-name hints in the grocery system prompt, but keep prices, sizes, and item IDs tool-only; then tell the judge to score pre-tool claims against the prompt-visible KB and post-tool claims against tool results/full KB -> This keeps the model aware of what exists without letting it answer catalog-detail questions from prompt memory, and it prevents bouquet-style grounding penalties based on hidden catalog rows.**
- **Conversation-bench rehydrate can false-fail late state summaries by using model history instead of golden prior context -> On `conversation_bench --rehydrate` runs, manually verify long-range-memory summary turns against `turns.py` goldens when the judge says prior registrations/actions never happened -> In the March 12, 2026 run `runs/conversation_bench/20260311T235158_gpt-realtime-1.5_a3ca59b7`, turns 52/56/57/58 matched the benchmark goldens but were judged from earlier model-side tool misses despite `runtime.json` noting "golden prior context" per turn.**
- **Appointment-bench parallel-32 rehydrate no-VAD can still be artifact-clean -> Treat `appointment_bench` runs with `--rehydrate --disable-vad --parallel 32` as scoreable when transcript/judged coverage stays `25/25` and `turn_taking_failures=[]`, then focus review on content errors rather than assuming concurrency-induced runtime corruption -> In the March 11, 2026 run `runs/appointment_bench/20260311T235157_gpt-realtime-1.5_2251e121`, the artifacts were clean and the remaining misses were turn 3's non-responsive post-tool utterance and turn 17's premature `end_session`.**
- **Judge grounding leniency needs concrete examples, not just abstract guidance -> If a recurring false fail comes from harmless present-tense commentary, add an explicit benchmark-shaped example to the judge prompt instead of only saying "be generous" -> In grocery-bench, abstract leniency rules did not clear turn 8 until the prompt explicitly said same-day-delivery follow-up language like "we should be good" should pass absent contrary time evidence.**
- **Prompt-filtered grocery KB shifts failures from fake tool misses to real follow-up issues -> After removing catalog facts from the grocery system prompt, first-discovery turns started calling `lookup_item` reliably, and the remaining judged misses clustered around over-clarifying quantity defaults, one unsupported same-day timing claim, and late-order edit/state mistakes -> Use the prompt-filtered setup when you want grocery-bench to measure actual retrieval plus workflow behavior instead of prompt-memory compliance.**
- **Prompt-facing KB should not include tool-retrievable catalog facts -> In grocery-bench, keep delivery/store-policy sections in `system_instruction` but strip product catalog and confusable-item tables from the prompt-facing KB -> Prevents first-discovery item turns from being answerable out of prompt memory while still allowing direct answers for store-policy questions.**
- **Judge CLI can silently drop benchmark KB context -> Always pass `knowledge_base.txt` into OpenAI/Claude judge calls, not just expected turns and system prompt fragments -> Prevents KB-grounding false fails where the judge treats benchmark facts as unsupported because they never reached the model prompt.**
- **Grocery-bench turn 8 same-day-delivery answers can be falsely marked ungrounded -> When `openai-v4-rehydrated-tool-results-in-prompt` returns `kb_grounding=false` on turn 8, verify `benchmarks/grocery_bench/data/knowledge_base.txt` before treating it as a model hallucination -> In the March 11, 2026 run `runs/grocery_bench/20260311T165504_gpt-realtime-1.5_a14b9360`, the assistant's "before 2 PM" and "2-4 hour window" answer matched the KB exactly, so the miss was judge-side or prompt-packaging-related rather than a real grounding failure.**
- **`--skip-turn-taking` still renders all-pass turn-taking summaries -> Treat `openai_summary.json` / judge console `turn_taking=25/25` on skip-turn-taking runs as a display artifact and rely on the flag plus `turn_taking_failures=[]`, not the pass count, when deciding whether turn-taking was actually evaluated -> Prevents rehydrated no-VAD reviews from accidentally treating forced-pass turn-taking fields as meaningful metrics.**
- **Tool-use scoring needs explicit first-discovery vs state-reuse guidance -> Add per-turn `tool_use_guidance` and surface it in the judge prompt so lookup-required catalog introductions stay strict while later turns can reuse already-established item facts without redundant lookups -> Prevents grocery-style evals from penalizing efficient session-state reuse and makes the intended contract legible to the judge.**
- **Realtime tool continuation needs two server-side gates -> Wait for both `conversation.item.added`/`conversation.item.done` on the injected `function_call_output` item and the prior `response.done` before sending deferred `response.create` -> Prevents missing a fast tool-output ack or fast response completion while the websocket send is still in flight, which can otherwise leave the continuation racing stale conversation state.**
- **Appointment-bench LLMAJ KB false fails are not stable under repeated replay -> When `gpt-5.2` marks appointment-bench turns 9 or 23 as `kb_grounding=false`, verify with repeated same-prompt replays before changing GT or prompt wording -> In a March 11, 2026 experiment on run `runs/appointment_bench/20260311T100636_gpt-realtime-1.5_448966ba`, replaying the current rehydrated LLMAJ prompt for turns 9 and 23 10 times at `reasoning_effort=none` and 10 times at `low` produced `kb_grounding=true` on 20/20 runs for both turns; `low` occasionally added a separate turn-23 completeness/state penalty, but never reproduced the original grounding miss.**
- **Rehydrated runtime summaries can overstate completion after a failed final turn -> Verify `transcript.jsonl` and judged-row coverage even when `runtime.json` says `failed_turns=[]` and all turns succeeded -> In the March 11, 2026 run `runs/event_bench/20260311T100636_gpt-realtime-1.5_0ba2711a`, `runtime.json` reported 29/29 success, but `transcript.jsonl` and `openai_judged.jsonl` stopped at turn 27 after turn 28 hit `no close frame received`, `Error connecting: [Errno 8] nodename nor servname provided, or not known`, and `session was not ready before rehydration seeding`.**
- **Rehydrated reconnect fallback must preserve tool results -> Mirror `function_call_response` payloads into reconnect history and render them back into the fallback prompt -> Prevents fresh-session retries from losing benchmark state after websocket reconnects.**
- **Grocery-bench rehydrate can complete cleanly while failures stay mostly model-side -> Treat `grocery_bench --rehydrate --disable-vad` runs with full `30/30` transcript coverage, `30/30` judged rows, and `turn_taking_failures=[]` as scoreable, then bucket remaining `gpt-realtime-1.5` misses into missed lookup/tool-completeness turns, pre-order state confusion, and post-order edit mistakes instead of runtime artifacts -> In the March 11, 2026 run `runs/grocery_bench/20260311T100643_gpt-realtime-1.5_364e7750`, the artifacts were clean, but turns 2/5/17/20/23 missed required lookup completeness, turn 19 wrongly claimed it could not edit the in-progress order yet, and turns 26-27 mishandled the post-order modification logic.**
- **Manual no-VAD retries need explicit state reset -> Clear OpenAI Realtime manual commit/response flags before replaying a turn's audio on retry -> Prevents retried `--disable-vad` turns from being dropped and mis-scored as model failures because duplicate stop events are ignored.**
- **Appointment-bench rehydrate skip-turn-taking needs manual bucket correction -> On `appointment_bench` rehydrated `gpt-realtime-1.5` runs, treat raw `kb_grounding` fails on turns 9 and 23 as likely judge grounding mistakes and subtract the expected closing `end_session` turn before interpreting `MODEL_ENDED_SESSION` counts -> In the March 11, 2026 run `runs/appointment_bench/20260311T100636_gpt-realtime-1.5_448966ba`, the artifacts were clean `25/25`, all required tool turns were on time, the only clear real model errors were turn 1's missed Thursday schedule constraint and turn 17's premature session close, while turns 2/9/23 were better explained by strictness or judge-side grounding issues.**
- **Event-bench quote and availability turns can under-answer despite same-turn tool results -> Turns 1, 6, 9, 14, 20, and 23 stop at "let me check/get a quote" even though `function_call_response` already contains the venue/price answer and later user turns presuppose that answer -> Treat these turns as GT issues before blaming models that answer the quote or availability directly.**
- **Some closing goldens add unsupported post-close narration -> `conversation_bench` turn 74 invites more help after `end_session`, and `event_bench` turn 28 plus `grocery_bench` turn 29 add confirmation-sending claims with no supporting tool or KB evidence -> Do not treat those extras as required behavior when triaging end-of-conversation failures.**
