## Recent Learnings

- **Event-bench rehydrate can now complete cleanly with VAD off -> Treat a full `event_bench --rehydrate --disable-vad` transcript as scoreable if transcript coverage reaches turns 0-28 and `runtime.json` shows `mode: "rehydrated"` with `disable_vad: true`, then bucket remaining misses as model-side tool/state errors rather than runtime artifacts -> A March 10, 2026 `gpt-realtime-1.5` run at `runs/event_bench/20260310T124029_gpt-realtime-1.5_904633d8` finished 29/29 turns and the judged failures clustered around missed tool commits, fabricated booked-state claims, and one arithmetic error, not transport instability.**
- **Product-bench rehydrate is unstable with VAD enabled -> Run `product_bench` rehydrated sweeps with `--disable-vad` or expect post-turn-0 active-response / websocket-timeout failures -> A March 10, 2026 `gpt-realtime-1.5` run failed immediately with `conversation_already_has_active_response` and keepalive ping timeouts, while the same benchmark completed 31/31 turns under `--rehydrate --disable-vad`.**
- **Manual OpenAI Realtime rehydrate must use current client-event wire shapes -> Normalize seeded assistant history items to assistant `output_text` content and avoid hand-built `response.create` payloads unless you have freshly verified the exact field names -> A March 10, 2026 live `event_bench` smoke run rejected assistant `content.type=\"text\"` during `conversation.item.create`, which showed the safe path here is to reuse current Pipecat/OpenAI event models for session-history seeding instead of relying on stale remembered wire shapes.**
- **`response.create.input` is syntactically valid but not a live-session substitute -> Prefer `conversation.item.create` seeding for prior-turn rehydration and treat `response.create.input` only as a per-response prompt override -> On March 10, 2026 `event_bench`, correct tool calls still drifted on post-tool continuation when prior turns were supplied only through `response.input`, confirming that lexical replay did not recreate accrued same-session audio/tool state.**
- **Event-bench run labels can be misleading -> Check `runtime.json.mode` and `disable_vad` before treating a folder as a normal-mode control -> Several March 2026 `event_bench` runs that looked like fresh normal repros were actually `mode: "rehydrated"` with manual turn handling, which can invalidate baseline comparisons.**
- **Tool-turn assistant preambles are preserved in transcripts -> Check `TurnGate._turn_transcript_parts` before assuming a pre-tool or retry preamble was dropped -> `TTSStoppedAssistantTranscriptProcessor` flushes each spoken segment and `TurnGate` joins multiple parts for one turn, so multi-segment tool turns like the normal-mode turn 9 retry are written intact into `assistant_text`.**
- **Realtime rehydration replays goldens, not actual prior model wording -> Treat `--rehydrate` as a benchmark-controlled conversation shape, not a byte-for-byte replay of the original assistant/tool-turn transcripts -> The rehydration path seeds prior turns from benchmark `golden_text` plus required tool calls/results, so differences from normal mode can come from the replayed history shape as well as model variance.**
- **Event-bench post-tool fallback is not pre-tool speech contamination -> Verify the first response emits only the function call and no assistant audio/text before `function_call_output`, then attribute the contradiction to the post-tool continuation itself -> In the March 9, 2026 `event_bench` rehydrate reproductions, turns 1/9/23 did not begin speaking until after the queued tool result was sent, which rules out a double-sampled or already-spoken fallback explanation.**
- **Event rehydrate turns can fail after correct tools succeed -> Bucket successful-tool plus fallback-language turns as post-tool grounding/commit errors, not retrieval misses -> In `event_bench` rehydrated `gpt-realtime-1.5` runs, turns 1, 9, and 23 called the expected tool successfully but answered as if availability/quote lookup had failed, so the decisive evidence is the contradiction between `tool_results` and the assistant text.**
- **`--skip-turn-taking` still emits placeholder turn-taking scores -> Treat `turn_taking` as excluded metadata only when `turn_taking_failures=[]` and every judged row has `turn_taking=true` -> Prevents content-only rehydrated reviews from accidentally counting the placeholder dimension as a meaningful pass/fail signal.**
- **Rehydrated watchdog logs can mislabel the affected turn -> Verify empty-response turns from `transcript.jsonl` and judged rows, not only the live watchdog stdout -> Prevents pipeline-log turn-number drift from being mistaken for a model/content failure on the wrong datapoint.**
- **Parallel rehydrate runs can finish with a hidden transcript gap after a transient realtime 502 -> Check transcript turn coverage before judging, then rerun missing turns with `--only-turns` and merge the recovered rows -> In `product_bench`, a single `HTTP 502` on one rehydrated OpenAI Realtime turn left the parent run hanging after the other turns completed, but the run was recoverable without rerunning the full benchmark.**
- **Grocery rehydrate failures can come after correct tool calls -> Bucket successful-tool plus failure-language turns as post-tool grounding errors, not retrieval misses -> In `grocery_bench` rehydrated runs, `gpt-realtime-1.5` can call `lookup_item` / `process_order` / `verify_details` correctly and then answer as if the tool failed, so the decisive evidence is the contradiction between `tool_results` and assistant text.**
- **Manual rehydrate tool continuation can drop the active user turn -> Re-include the current committed user audio item in post-tool `response.create` input on the same session -> Without that, rehydrated OpenAI Realtime turns can execute the first tool correctly and then drift into `end_session` or incoherent follow-up because the continuation sees tool output without enough reminder of the user request.**
- **Appointment tool results can be script-shaped rather than semantically trustworthy -> Check `tool_calls` name/args before trusting `tool_results`, especially on wrong-tool turns -> In normal-mode `appointment_bench`, a model can call the wrong function on a tool-required turn and still receive that turn's scripted `function_call_response` payload, so the result object alone can make a bad action look superficially valid.**
- **Rehydrated turn-taking analysis is misleading -> Re-judge `--rehydrate` runs with `--skip-turn-taking` and preserve any raw turn-taking summaries separately -> Transcript rows accumulate across all rehydrated turns, but the realtime pipeline writes a single `conversation.wav` in the run root, so the audio analyzer does not see a true full-conversation waveform.**
- **Benchmark contract mismatches can look like model regressions -> Check system prompt, KB, and golden turns together before calling a failure real -> In `conversation_bench`, some goldens require answers the system prompt discourages (for example directions/hotels and explicit counts), and in `appointment_bench` some judgments over-index on the golden text instead of allowing KB-supported extra facts.**
- **Consistency result folders may be placeholders -> Verify `metadata.json` and `run_results.csv` before trusting any `*_consistency_*` directory, and rebuild comparison outputs from judged run folders when those files are missing -> Prevents empty shell directories in `results/` from being mistaken for usable repeated-run artifacts.**
- **Event bench failures can cascade after a missed commit step -> Separate the first real workflow miss from later readback/deposit/state failures before counting root causes -> In `event_bench`, once the model skips `book_event` or `update_event`, later turns often fail because the model keeps reasoning from an uncommitted or hypothetical configuration rather than because each later answer is a new independent mistake.**
- **Normal-mode transcript gaps can break judging -> Check transcript turn coverage before running `audio-arena judge`, and if a realtime run dropped a turn, judge only the present turns or fall back to manual review -> Prevents the OpenAI judge from failing hard on missing transcript rows and explains downstream cross-turn contamination artifacts.**
- **Realtime grocery tool results are canned by turn -> Treat transcript `tool_results` as evidence that a tool was called, not proof the args/workflow were semantically correct -> In normal-mode grocery runs, the pipeline returns the turn's scripted `function_call_response` whenever a tool is invoked, so late-turn `success` payloads can coexist with wrong arguments or wrong-order actions.**
- **Event-bench post-tool fallback language can survive every client-side continuation fix -> After eliminating out-of-band call IDs, waiting for server acknowledgement of `function_call_output`, duplicating raw tool JSON into continuation instructions, and injecting an extra same-session `conversation.item.create` system message that restates the tool result, remaining "lookup failed" continuations on event-bench turns 1/9/23 are model-side grounding failures in `gpt-realtime-1.5`, not pipeline ordering bugs -> Prevents wasting more time on transport-level fixes once the continuation is already fully conversation-scoped and acknowledged.**
- **Transcript-derived rehydration still triggers the same post-tool failures -> If turns 1/9/23 still fail after replacing benchmark-golden rehydration items with clean normal-run transcript-derived items, the causal delta is not golden wording drift; it is the fresh-session injected-history path itself -> In `event_bench`, `gpt-realtime-1.5` does not treat prior turns inserted via rehydration as equivalent to history accrued live in-session for post-tool continuation grounding.**
- **Prefix-only `response.input` on post-tool rehydrate continuation changes the failure mode, not the outcome -> Treat `response.input` replay of only prior golden items as a diagnostic branch, not a fix -> On March 10, 2026 `event_bench`, turn 1 lost the explicit fallback language but collapsed into a generic trailing sentence, turn 9 ignored the successful quote and re-asked for the music choice, and turn 23 emitted `end_session`.**
- **Re-sending the committed audio item in post-tool rehydrate continuation backfires -> Do not include the current `item_reference` in the follow-up `response.create` after `function_call_output`; it can trigger duplicate tool calls and watchdog-driven turn requeue loops -> On March 10, 2026 `event_bench` turns 1/9/23, adding `response.input` with the committed audio item caused a second `search_venues` call on turn 1 and later `Re-queuing audio for turn 0` instead of fixing the post-tool fallback language.**
- **Default-VAD event rehydrate can hang after a successful tool call -> Prefer `--disable-vad` for full `event_bench --rehydrate` runs until the active-response race is fixed, and treat partial default-VAD runs as runtime artifacts rather than scoreable results -> A fresh March 10, 2026 run at `runs/event_bench/20260310T120938_gpt-realtime-1.5_d3ec15c7` hit `conversation_already_has_active_response` after turn 1's `function_call_output`, then fell into watchdog retries and only wrote transcript turns 0-2.**
- **OpenAI Realtime handshake timeouts can leave rehydrate runs looking alive but transcript-frozen -> If `transcript.jsonl` stops advancing while `run.log` still grows, check for `timed out during opening handshake` and `session was not ready before rehydration seeding`, then treat the run as a runtime artifact instead of waiting indefinitely -> A fresh March 10, 2026 `assistant_bench --rehydrate --disable-vad` run at `runs/assistant_bench/20260310T121111_gpt-realtime-1.5_6357786c` stopped after turn 7 when turn 8 failed to open the websocket session and never resumed transcript writes.**
- **Appointment booking-ID omissions can be judge-only fails -> Treat `appointment_bench` booking confirmations that get the tool call, time math, and schedule right but omit `Appointment ID` as likely strictness issues unless the task explicitly depends on the ID -> In the March 10, 2026 rehydrated skip-turn-taking run `runs/appointment_bench/20260310T121056_gpt-realtime-1.5_2347e58a`, turn 12 failed only because the spoken confirmation omitted `APT-002` even though the booking succeeded and the before-5:00 reasoning was correct.**
